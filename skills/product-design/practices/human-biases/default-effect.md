## Principle
Users overwhelmingly stick with pre-selected options — making defaults the single most powerful design decision in any interface that involves configuration or choice.

## Why It Matters
Research consistently shows that the vast majority of users (often 70-90%) never change default settings, regardless of whether those defaults serve their interests. This happens because changing defaults requires effort, users interpret defaults as implicit recommendations, and deviating from them triggers uncertainty. This means defaults are not neutral — they are decisions made on behalf of users. A default set to "public" means most profiles will be public. A default set to "weekly emails" means most users get weekly emails. Defaults are policy.

## Application Guidelines
- Set defaults to the option that benefits the most users, not the option that benefits the business most
- For privacy and security settings, default to the most protective option
- Audit all defaults regularly — they should reflect current best practices and user research, not historical accidents; a default that was right for early adopters may not suit a mainstream audience
- When defaults carry significant consequences (data sharing, billing), require explicit opt-in rather than relying on opt-out
- Use smart defaults that adapt based on user context (locale, device, usage patterns) rather than one-size-fits-all
- Pre-select recommended plan tiers, but make the selection visually obvious so users know a choice has been made on their behalf
- Provide "recommended" or "most popular" labels on defaults so users who do review options understand why this one was pre-selected

## Anti-Patterns
- Pre-checking newsletter signup boxes or data-sharing consents
- Setting default sharing permissions to "public" or "everyone" when most users want privacy
- Burying the option to change defaults deep in settings while the default benefits the platform
- Using confusing double-negatives in opt-out language ("uncheck to not receive...")
