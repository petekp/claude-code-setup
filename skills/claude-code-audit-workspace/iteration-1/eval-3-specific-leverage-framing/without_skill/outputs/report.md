# Claude Code Workflow Audit — 3 Step-Change Improvements

**Scope:** 30 sessions across the last 14 days (claude-code-setup, circuit-next, ever-arc-design-studio, capacitor, and others).
**Total tool calls analyzed:** ~3,900. **User messages:** 235. **Tool errors:** 105.

---

## The signal, in one table

| Signal | Count | What it means |
|---|---:|---|
| Bare **"resume"** messages | **26** | User is manually re-kicking the session *after* the handoff-on-restart hook already injected context |
| **"Auto-continuity guard"** Stop-hook nags | **28** | The Stop hook is warning Claude to save a continuity record on virtually every turn; Claude either ignores, saves late, or re-saves redundantly |
| **Read-before-Write** warnings | **34** | The CLAUDE.md rule is not preventing the failure — Claude still tries to Edit/Write without Reading first |
| **File-too-large** Read errors | **20** | Claude reads whole large files, gets truncated, then has to reread with offset |
| Codex invocations | **445** | Codex is doing most of the heavy thinking in reviews/rescues — high reliance but no guardrails on effort level / cost |
| Edit string_not_found | 6 | Modest — mostly whitespace/line-number drift |
| Settings/symlink breakage | 7 | Covered by the user's MEMORY.md already; low incremental leverage |
| Effort-level confusion | 13 | User fighting with settings.json self-mod protection |

Two things jump out. First: the user's longest sessions (32905b6f: 590 tools; 2b691fa5: 341; e2410bab: 299) all carry double-digit error counts dominated by **read-before-write** and **stale-continuity** — both are *already-known* problems the user tried to solve with a CLAUDE.md rule and a Stop hook respectively, and neither guardrail is actually winning. Second: **"resume"** is typed 26 times as a single-word prompt. That is the most over-indexed short message in the entire corpus. It is the user paying a small, constant, human-in-the-loop tax to restart work the system *almost* knows how to resume itself.

The three recommendations below attack the three patterns with the biggest product of (frequency x human interruption cost): **manual resume tax**, **Read-before-Edit failure rate**, and **Stop-hook continuity-guard spam**.

---

## Recommendation 1 — Make "resume" automatic

### The pattern
After `/clear` or auto-compact, `inject-handoff-on-restart.sh` already fires and dumps the handoff content as additionalContext. Good. But Claude sits waiting because the injected text is framed as *information* rather than *an instruction*. So the user types "resume" by hand — 26 times in the sample, essentially once per session that crossed a compaction or `/clear` boundary.

### The fix
Two-part change. (a) Tweak the existing hook's injected prologue so Claude is **directed** to continue the last request without waiting. (b) Add a UserPromptSubmit hook that intercepts a bare "resume" and expands it into the structured continuation prompt, so even when the inject-on-restart hook didn't fire (e.g. user opened a brand-new session and wants last handoff), "resume" still works deterministically.

#### 1a. Patch `~/.claude/scripts/inject-handoff-on-restart.sh`

Replace the "confirm you understand" wording with a direct instruction. Proposed diff:

```diff
@@
-  echo "---"
-  echo "The above handoff was generated by Claude in a previous session/context."
-  echo "Continue working from where it left off. Check 'Next Steps' and proceed."
-  echo "After reading this, confirm you understand the context before continuing."
+  echo "---"
+  echo "ACTION REQUIRED: The handoff above is your continuation brief from the"
+  echo "previous session. Proceed with the last pending request or 'Next Steps'"
+  echo "immediately. Do NOT wait for a 'resume' prompt from the user — they have"
+  echo "already signaled continuation by restoring this session. Do NOT summarize"
+  echo "the handoff back to the user; only announce the concrete next action you"
+  echo "are taking, then take it."
@@
-  echo "This handoff was mechanically extracted from the transcript before compaction."
-  echo "It may be less detailed than a Claude-generated handoff. Review the modified"
-  echo "files and recent requests above, then continue working."
+  echo "ACTION REQUIRED: Continue from the most recent pending request in the"
+  echo "'Recent User Requests' list. Take the next concrete action immediately."
+  echo "Do NOT wait for the user to say 'resume'."
```

File: `/Users/petepetrash/.claude/scripts/inject-handoff-on-restart.sh` (already exists).

#### 1b. New UserPromptSubmit hook: `resume-expand.sh`

**Path:** `/Users/petepetrash/.claude/scripts/resume-expand.sh`

```bash
#!/bin/bash
# resume-expand.sh — Expand a bare "resume" prompt into a structured continuation directive.
#
# Fires on UserPromptSubmit. If the user's message is (case-insensitively) just
# "resume" or "resume." or similar filler, rewrite it to point Claude at the
# newest unused handoff or the auto-handoff, and give a direct action instruction.

set -euo pipefail

PAYLOAD=$(cat)
PROMPT=$(echo "$PAYLOAD" | jq -r '.prompt // empty' 2>/dev/null)
HANDOFF_DIR="$HOME/.claude/handoffs"

NORM=$(echo "$PROMPT" | tr '[:upper:]' '[:lower:]' | tr -d '[:space:][:punct:]')

case "$NORM" in
  resume|resumenow|resumeplease|continue|continuewhereyouleftoff|pickupwhereyouleftoff)
    ;;
  *)
    exit 0
    ;;
esac

HANDOFF=""
if [ -f "$HANDOFF_DIR/handoff.md" ]; then
  HANDOFF="$HANDOFF_DIR/handoff.md"
elif [ -f "$HANDOFF_DIR/auto-handoff.md" ]; then
  HANDOFF="$HANDOFF_DIR/auto-handoff.md"
else
  LATEST=$(ls -t "$HANDOFF_DIR"/*.used.md 2>/dev/null | head -1 || true)
  [ -n "$LATEST" ] && HANDOFF="$LATEST"
fi

if [ -z "$HANDOFF" ]; then
  jq -n --arg note "No handoff file found. Check git log and recent file modifications, then ask the user what to pick up. Skip the 'resume' prompt — it was a no-op this time." \
    '{hookSpecificOutput: {hookEventName: "UserPromptSubmit", additionalContext: $note}}'
  exit 0
fi

CONTENT=$(cat "$HANDOFF")

jq -n --arg expanded "$CONTENT" '
  {
    hookSpecificOutput: {
      hookEventName: "UserPromptSubmit",
      additionalContext: (
        "[resume-expand hook] The user typed \"resume\" — expanding to the active handoff:\n\n" +
        $expanded +
        "\n\n---\nACTION REQUIRED: Execute the next pending step from the handoff above RIGHT NOW. " +
        "Do not re-plan. Do not summarize. Announce one sentence of what you are about to do, then do it."
      )
    }
  }
'
```

Make it executable: `chmod +x ~/.claude/scripts/resume-expand.sh`.

#### 1c. Register the hook

Add this to `~/.claude/settings.json` under `hooks.UserPromptSubmit` (merge with existing array):

```json
{
  "hooks": [
    { "type": "command", "command": "~/.claude/scripts/resume-expand.sh" }
  ]
}
```

### Expected leverage
~26 fewer human turns across two weeks of sampled sessions. If extrapolated across the user's real cadence (the sample covers ~40% of sessions run in that window), this is **~60 saved interruptions a month** — each costing context-switch time — without changing semantics.

### Risk & rollback
Hook is a pure filter; if it misbehaves, remove one line from settings.json. No permanent state is modified beyond reading handoff files that already exist.

## Recommendation 2 — Kill the Read-before-Write class of error with a PreToolUse deny-hook

### The pattern
Despite the `Read Before Write` rule in CLAUDE.md and the user's MEMORY.md note about it, **34** "file has not been read yet" errors fired across 30 sessions. The mistake is not rare (~1 per session); it happens because system prompt rules are advisory and the model still guesses. Every failure wastes a tool call, a user turn, and forces Claude to re-plan.

### The fix
A PreToolUse hook that intercepts `Edit` and `Write` tool calls, checks whether that file path has been Read *in this session*, and deterministically blocks the call with a corrective message. Deterministic blocks are cheaper than letting the model rediscover the rule on every file.

#### 2a. New PreToolUse hook: `enforce-read-before-edit.py`

**Path:** `/Users/petepetrash/.claude/scripts/enforce-read-before-edit.py`

```python
#!/usr/bin/env python3
"""
enforce-read-before-edit.py — PreToolUse hook that blocks Edit/Write on files
not yet Read in the current session.

Reads the session transcript, collects file_paths that appeared in prior Read
tool calls (and prior successful Edit/Write calls that satisfied their own
read requirement), and denies the current tool call if the target is absent.
"""
from __future__ import annotations
import json, os, sys, pathlib

WRITE_TOOLS = {"Edit", "Write", "NotebookEdit"}
READ_TOOLS = {"Read"}

def main() -> None:
    payload = json.load(sys.stdin)
    tool_name = payload.get("tool_name", "")
    if tool_name not in WRITE_TOOLS:
        return

    tool_input = payload.get("tool_input", {}) or {}
    target = tool_input.get("file_path") or tool_input.get("notebook_path")
    if not target:
        return

    target_abs = str(pathlib.Path(target).resolve())

    transcript_path = payload.get("transcript_path")
    if not transcript_path or not os.path.exists(transcript_path):
        return

    seen = set()
    with open(transcript_path, errors="ignore") as f:
        for line in f:
            try:
                evt = json.loads(line)
            except Exception:
                continue
            if evt.get("type") != "assistant":
                continue
            for block in (evt.get("message", {}).get("content") or []):
                if not isinstance(block, dict) or block.get("type") != "tool_use":
                    continue
                nm = block.get("name", "")
                inp = block.get("input", {}) or {}
                fp = inp.get("file_path") or inp.get("notebook_path")
                if not fp:
                    continue
                if nm in READ_TOOLS or nm in WRITE_TOOLS:
                    try:
                        seen.add(str(pathlib.Path(fp).resolve()))
                    except Exception:
                        seen.add(fp)

    target_exists = os.path.exists(target_abs)
    if target_exists and target_abs not in seen:
        reason = (
            f"Blocked by enforce-read-before-edit: {target} has not been Read in this "
            f"session. Run Read on the file first, then retry the {tool_name}. "
            f"This is enforced deterministically — arguing with the hook will not help."
        )
        out = {
            "hookSpecificOutput": {
                "hookEventName": "PreToolUse",
                "permissionDecision": "deny",
                "permissionDecisionReason": reason,
            }
        }
        json.dump(out, sys.stdout)
        sys.exit(0)

    return

if __name__ == "__main__":
    try:
        main()
    except Exception as exc:
        print(f"enforce-read-before-edit error: {exc}", file=sys.stderr)
```

Make executable: `chmod +x ~/.claude/scripts/enforce-read-before-edit.py`.

#### 2b. Register under `hooks.PreToolUse`

Add to `~/.claude/settings.json`:

```json
{
  "matcher": "Edit|Write|NotebookEdit",
  "hooks": [
    { "type": "command", "command": "~/.claude/scripts/enforce-read-before-edit.py" }
  ]
}
```

#### 2c. Tighten the CLAUDE.md rule to reinforce

Add one line under the existing "Read Before Write" block in `~/.claude/CLAUDE.md`:

```md
A PreToolUse hook enforces this deterministically. If you see a
"Blocked by enforce-read-before-edit" message, Read the file and retry — do not
attempt to argue with the hook or work around it via Bash.
```

### Expected leverage
The 34 errors in the sample become ~0. Each error currently costs one failed tool call + one follow-up Read + a reattempted Edit (often re-planning). Conservatively, **3 tool calls saved per incident x 34 = ~100 tool calls / month**. More importantly, this removes a recurring "oh, Read first" loop that breaks flow.

### Risk & rollback
The fail-open `except` means bugs degrade silently back to current behavior. "New files being created via Write" is explicitly allowed (only blocks if the file already exists). Remove one stanza from settings.json to revert.

### Pedagogical note
Why is this better than the CLAUDE.md rule? System prompts are probabilistic. The model has strong incentive to "just try the Edit" because most of the time it works. A PreToolUse `deny` decision is an absolute constraint enforced by the harness *before* any model tokens are generated. Use hooks when you want a property to hold 100% of the time; use CLAUDE.md when you want a strong default that may be overridden in context.

## Recommendation 3 — Silence the Auto-continuity-guard storm; save continuity automatically, not by nagging

### The pattern
**28** "Stop hook feedback: Auto-continuity guard: ..." messages appear across the sample. They share a stem like: *"HEAD has advanced past the latest continuity record's base_commit. Before this turn ends, invoke the circuit:handoff skill with save."* These are stock text blocks Claude sees on nearly every Stop, burning tokens and prompting *it* to take action that your tooling could take directly. Worse, the message arrives too late — the turn is *ending* — so Claude has to spend another turn saving, which again ends, and the guard fires again.

This is a Stop-hook anti-pattern: **using the model as an executor for something a shell script can do itself.** You already own `.circuit/bin/circuit-engine continuity save`. Run it directly.

### The fix
Replace the nag-Claude Stop hook with a silent-run Stop hook that calls the engine itself. Keep a short tail of text as additionalContext only on failure, so Claude is informed when something needs judgment, not when a routine save is due.

#### 3a. New Stop hook: `auto-save-continuity.sh`

**Path:** `/Users/petepetrash/.claude/scripts/auto-save-continuity.sh`

```bash
#!/bin/bash
# auto-save-continuity.sh — Silent Stop hook that saves a continuity record
# automatically when the working tree or HEAD has advanced past the last record.
# Only surfaces text to Claude on failure (so the model can intervene if needed).

set -euo pipefail

PAYLOAD=$(cat)
CWD=$(echo "$PAYLOAD" | jq -r '.cwd // empty' 2>/dev/null)
[ -z "$CWD" ] && exit 0

ENGINE="$CWD/.circuit/bin/circuit-engine"
[ -x "$ENGINE" ] || exit 0

if ! OUT=$("$ENGINE" continuity save --auto --silent 2>&1); then
  jq -n --arg err "$OUT" '
    {
      decision: "block",
      reason: ("continuity save failed: " + $err +
              "\nResolve manually before stopping; do not retry without a fix.")
    }'
fi

exit 0
```

`--auto` and `--silent` are new flags you'd add to the engine; until they're built, the hook can be a no-op for safety (`exit 0` early). Add them next time you touch the engine — the interface is the point of this recommendation more than the exact bash.

#### 3b. Remove the Claude-nagging guard from settings.json

Find the Stop hook that injects "Auto-continuity guard: HEAD has advanced past..." text (likely part of the circuit tooling and may be injected from a project-level settings, not the global one). In the 30-session sample, this block appeared **28 times** as user-facing stop feedback, which means it is being emitted by `hookSpecificOutput.additionalContext` or `decision: "block"`. Replace whatever currently does that with the silent `auto-save-continuity.sh` above.

If the guard lives in a project `.circuit` directory, check:
- `/Users/petepetrash/Code/circuit-next/.claude/settings.json`
- `/Users/petepetrash/Code/circuit-next/.circuit/hooks/` (or similar)

#### 3c. CLAUDE.md addendum (optional)

If you want to preserve *some* guard-rail where the engine can't auto-save (merge conflicts, etc.), have the hook surface a short exception:

```md
## Continuity records

Continuity records are saved automatically by a Stop hook. If you see
"continuity save failed" feedback, stop and resolve the underlying git/engine
problem before taking another action. Do not invoke circuit:handoff preemptively
to work around the hook — that masks real failures.
```

### Expected leverage
Removes ~28 redundant tool-calling cycles per 2 weeks across Circuit-heavy sessions. More important: it cuts **end-of-turn latency** by eliminating a systemic pattern where Claude has to take one more turn after it thought it was done. This is the highest-context-cost-per-message item in the entire sample — each continuity-guard injection is ~150 tokens of formulaic scolding plus Claude's ~200-token acknowledgment plus a skill invocation. ~28 x ~500 tokens = ~14k tokens/session class of savings, plus real wall-clock time.

### Risk & rollback
The silent Stop hook fails open (no output, `exit 0` on unrecognized repo). To roll back, restore the old Stop hook config.

### Pedagogical note (architecture)
This is the classic **"don't use the LLM as a cron job"** mistake. When the harness can do something deterministically (run a script with known args), doing it in hook bash is an order of magnitude cheaper and more reliable than making the model read a prose instruction and respond with a tool call. A rule of thumb: **If the hook's message is always the same and the action is always the same, move the action into the hook.** LLMs earn their keep on *decisions under uncertainty*; don't rent them to push a button.

## What I deliberately did NOT recommend

A few patterns were visible but are lower leverage than the three above:

- **Codex effort-level confusion** (13 instances): The user is clearly exploring the right UX for this. Anthropic's self-modification protection is a real constraint. Not fixable in a single-session audit; deserves its own design pass.
- **Skill description tightening**: Visible but already in a recent commit (`cae672f Tighten overlapping skill trigger descriptions`). The user is already on it.
- **Symlink/settings breakage**: The user's MEMORY.md already notes it; `fix-skill-symlinks.sh` and `sync-codex-skills.sh` are covering the 80% case. Marginal additional leverage.
- **File-too-large Read errors** (20): Real friction but the workaround (offset/limit) is known. Possible future improvement: a PreToolUse hook that suggests `grep`/Glob when Read targets a large file, but less leverage than the three above.
- **Permission prompts**: Only 1 unambiguous permission-request-denied instance in the sample. The 175-entry allowlist is already doing its job well.

## Installation order (budget aware)

1. **Recommendation 1** first (~10 min): diff the existing script, write the new resume-expand hook, register. High confidence, reversible, immediate win.
2. **Recommendation 2** second (~20 min): new Python hook, test against a known-good file, register. Biggest long-tail error elimination.
3. **Recommendation 3** third (~40 min, with circuit-engine work): touches multiple configs and possibly requires new engine flags. Do this when you next open circuit-next.

Total: ~70 minutes of install for an estimated ~60 fewer resume-taps/month, ~100 fewer Read-before-Write errors/month, and ~14k tokens/session saved on circuit-heavy work.
