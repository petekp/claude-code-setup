# Implementation Options: Product-Design Skill Improvement

## Option 1: Compiled Index with Deduplication Sweep

The conservative, infrastructure-first approach. Fix what is broken and add machinery for smarter retrieval without reorganizing the file tree. The central idea: build a single `index.md` file that acts as a queryable manifest of every practice, with tags, priority tiers, and situation keywords compiled from the corpus. The model scans the index (~2K tokens) to pick 3-5 files instead of relying on the situation router's broken references or globbing entire directories.

- **Architecture shape:** Three-tier stays intact. SKILL.md shrinks (30 Essential Principles removed or collapsed to 15 highest-signal). A new `practices/index.md` sits between SKILL.md and the individual files, acting as a Tier 2.5 lookup table. The 25-category directory structure is untouched. Duplicate files are merged in-place (27 exact + ~15 semantic = ~40 files removed, net count drops to ~445).
- **Key seam:** `index.md` becomes the single source of truth for file-to-concept mapping. Every practice gets a one-line entry: filename, title, 3-5 tags, priority (core/situational/niche), and a 10-word "use when" clause. SKILL.md's situation router is rewritten to say "scan index.md for matching tags" instead of hardcoding filenames.
- **Retrieval mechanism:** Top-down, two-hop. SKILL.md triggers, model reads index.md (~2K tokens), model scans tags/keywords to find 3-5 relevant files, model reads those files. No globbing required for common situations. Glob remains as fallback for edge cases.
- **Scope of changes:** ~45 files deleted (duplicates). ~445 files get frontmatter enriched (tags, priority, applies_when). 1 new file created (index.md). SKILL.md rewritten (~60% of content changes). Total: ~490 files touched, but changes to practice files are frontmatter-only (scriptable).
- **Failure surface:** The index becomes a maintenance liability -- every new or renamed practice file requires an index update. If the index drifts from reality, retrieval degrades worse than the current glob fallback. Also, 2K tokens for an index scan is non-trivial overhead on every invocation.
- **Prerequisite changes:** Must identify and merge all duplicates first. Must define the tag vocabulary before enriching frontmatter (otherwise tags are inconsistent and useless). Must audit the situation router to catalog which references are broken vs. working.
- **Rollback cost:** Low. Index.md can be deleted, SKILL.md reverted. Frontmatter additions are additive (don't break anything if ignored). Merged duplicates are the only irreversible change, but the content is preserved in the surviving file.
- **Explicit disqualifiers:** If the model's context window cannot reliably process a 2K-token index to pick the right 3-5 files (i.e., if the index is too dense to parse accurately), this approach fails at its core mechanism. Also disqualified if maintenance scripting is not feasible (the index must be regenerated, not hand-maintained).

## Option 2: Intent-Based Restructure with Absorbed Theory

The aggressive reorganization approach. Replace the 25-category taxonomy with 7-9 intent-based top-level groups that match how design questions actually arise in development work. Theoretical categories (cognitive-psychology, human-biases, gestalt, behavioral-psychology, reading-psychology, gamification -- 39% of files) are dissolved: their content is absorbed into the applied categories where it is actionable. A practice about "Hick's Law" doesn't live in `cognitive-psychology/` -- it gets split into concrete guidance fragments inside `forms-and-input/`, `navigation/`, and `component-patterns/` where Hick's Law actually applies.

- **Architecture shape:** Completely new directory structure. ~7-9 intent directories like: `building-forms/`, `designing-layouts/`, `handling-data/`, `navigating-flows/`, `providing-feedback/`, `establishing-systems/`, `platform-specifics/`, `ai-and-emerging/`. Each directory contains 15-30 practice files that combine theoretical grounding with applied guidance. SKILL.md becomes a thin intent-matcher (~80 lines). No Essential Principles section -- the principles are woven into the practices that use them.
- **Key seam:** The theory-to-application absorption. Each theoretical principle gets "applied" by embedding its "Why It Matters" reasoning into the specific applied practices where it creates value. The standalone theoretical file ceases to exist. This means `hicks-law.md` becomes paragraphs inside `reduce-options-at-decision-points.md`, `progressive-form-disclosure.md`, and `navigation-menu-depth.md`.
- **Retrieval mechanism:** Single-hop directory routing. The user's intent maps directly to one directory. SKILL.md contains a concise intent table (~7-9 rows). Model globs one directory, reads 3-5 files. No index needed because each directory is small enough (15-30 files) to scan by title. Essentially the current mechanism but with directories that are right-sized for glob-and-pick.
- **Scope of changes:** Every file moves or is rewritten. ~191 theoretical files dissolved. ~295 applied files reorganized and enriched with absorbed theory content. ~40 duplicates eliminated in the process. Net file count: ~250-300 (down from 486). New coverage gap files added: ~30-50 (responsive, animation, dark mode, etc.). SKILL.md completely rewritten.
- **Failure surface:** The absorption process is high-risk. Theoretical principles often apply to 5+ categories -- deciding which applied file gets the theory requires judgment calls, and getting it wrong means the principle is effectively lost (buried in an unexpected file). Also, the reorganization invalidates any muscle memory or mental model the model has built for the current structure. And the scope is enormous -- touching every file means high probability of introducing new errors.
- **Prerequisite changes:** Must define the intent taxonomy before any file moves. Must create a mapping of every theoretical principle to its target applied practices. Must decide the "primary home" for principles that apply broadly (e.g., Fitts's Law applies to forms, navigation, and component design -- which directory owns it?).
- **Rollback cost:** Very high. This is a from-scratch restructure. Rolling back means restoring 486 files from git history. The absorbed theory is especially hard to undo because it is no longer a standalone artifact.
- **Explicit disqualifiers:** Disqualified if the team cannot invest 4-8 hours of focused work on the restructure (this is not a quick fix). Disqualified if the theoretical depth is valued as standalone reference material (some users may want to read "all Gestalt principles" as a category, which this approach destroys). Disqualified if the skill is actively used by multiple people who would be disrupted by the structural change.

## Option 3: Fix the Router, Shrink the Principles, Leave the Corpus Alone

The minimalist, highest-ROI approach. The diagnosis shows that the biggest failures are in the SKILL.md layer, not in the practice files themselves. The situation router has 42% broken references. The 30 Essential Principles waste ~1,500 tokens duplicating practice files. The practice files themselves are well-structured and well-sized (~520 tokens each). So: fix the seams, don't reorganize the corpus.

- **Architecture shape:** Same 25-category structure. Same 486 practice files (minus ~27 exact duplicates = ~459). SKILL.md gets a surgical rewrite: broken router references fixed, Essential Principles cut from 30 to 10 (the truly universal ones that apply to every interface decision), situation router expanded from 17 rows to ~25 (covering more of the 25 categories). Practice files untouched except duplicate removal.
- **Key seam:** The situation router table in SKILL.md. Every row gets verified file references (not guessed filenames). Each row gets 3-4 starter files that are confirmed to exist and are the highest-signal entry points for that situation. The router table becomes the single retrieval mechanism -- no index, no frontmatter queries.
- **Retrieval mechanism:** Same as today but with correct references. Model reads SKILL.md (~5K tokens, down from ~7K after principle reduction). Matches user situation to router row. Reads 3-5 files from the specified directory. Glob fallback for situations not in the router.
- **Scope of changes:** ~27 files deleted (exact duplicates). SKILL.md rewritten (~40% of content changes). No new files created. No frontmatter changes. No directory changes. Total: ~28 files touched.
- **Failure surface:** Minimal. The only risk is that the reduced Essential Principles section loses coverage for situations where no router row matches and the model falls back to the inlined principles. Also, without frontmatter enrichment or an index, there is no path to cross-category retrieval (e.g., a query that spans forms + accessibility cannot be routed to a single directory).
- **Prerequisite changes:** Must audit every router reference against actual filenames. Must rank the 30 Essential Principles to decide which 10 to keep. Must identify semantic duplicates (beyond the 27 exact) to decide whether any additional merges are warranted.
- **Rollback cost:** Trivially low. One file (SKILL.md) to revert, 27 deleted files to restore from git.
- **Explicit disqualifiers:** Disqualified if the core problem is practice file quality rather than routing accuracy. Disqualified if coverage gaps (missing domains like responsive, animation, dark mode) are considered higher priority than retrieval precision -- this approach adds zero new content. Disqualified if cross-category queries are a significant use case (this approach has no mechanism for them).

## Option 4: Layered Playbooks Over a Frozen Corpus

A different philosophy entirely: leave the 486 practice files as a frozen reference corpus and build a new retrieval layer on top. Instead of improving individual practice files or reorganizing directories, create 8-12 "playbook" files that are situation-specific compositions -- curated sequences of practices with synthesis commentary that tell the model exactly which files to read and in what order for common design tasks. The playbooks are the primary retrieval target; the practice files become a backing store that playbooks reference.

- **Architecture shape:** New `playbooks/` directory alongside `practices/`. Each playbook is a 300-500 token file that covers a common design situation (e.g., `playbooks/designing-a-form.md`, `playbooks/building-a-dashboard.md`, `playbooks/reviewing-existing-ui.md`). A playbook names 5-8 specific practice files to read, explains how the principles interact for that situation, and flags common mistakes. SKILL.md's situation router points to playbooks instead of raw practice directories. The 30 Essential Principles are replaced by a single "universal playbook" for situations that don't match a specific one.
- **Key seam:** Playbook-to-practice file references. Each playbook is a curated reading list with synthesis. The playbook is the unit of retrieval; individual practice files are read on the playbook's instruction. This creates a two-document retrieval pattern: read the playbook (~400 tokens), then read its referenced practices (~2,500 tokens for 5 files). Total: ~3K tokens per situation, well under the current ~7K.
- **Retrieval mechanism:** Top-down, two-hop with editorial curation. SKILL.md maps situations to playbooks (10-12 rows). Model reads one playbook. Playbook tells the model which practice files to read. The model never needs to glob, scan an index, or interpret frontmatter -- the playbook has already done the selection work.
- **Scope of changes:** 8-12 new playbook files created. SKILL.md rewritten to reference playbooks. 30 Essential Principles removed. ~27 duplicate practice files deleted. Practice file content untouched. Total: ~40 files touched (12 new, 1 rewritten, 27 deleted).
- **Failure surface:** The playbooks become a second layer of editorial content that must be maintained in sync with the practice files. If a practice file is renamed, moved, or deleted, every playbook referencing it breaks -- the exact same problem the current router has, but now distributed across 12 files instead of one table. Also, situations not covered by a playbook have no good fallback (the model must revert to globbing raw practice directories). And the playbook author must make good curation choices -- a bad playbook that references the wrong 5 practices is worse than no playbook at all.
- **Prerequisite changes:** Must identify the 8-12 most common design situations from actual usage patterns (not theoretical categories). Must audit practice files to select the 5-8 best for each playbook. Must decide the playbook format (reading list only? synthesis commentary? decision tree?).
- **Rollback cost:** Low. Delete the `playbooks/` directory, revert SKILL.md. Practice files are untouched.
- **Explicit disqualifiers:** Disqualified if the user's design queries are highly varied and rarely match pre-composed situations (playbooks only help for anticipated scenarios). Disqualified if the practice corpus is too low-quality to be a useful backing store (this approach assumes the practices are good, just poorly surfaced). Disqualified if 8-12 playbooks cannot cover enough situations to justify the editorial investment.

## Option 5: Tag-and-Query with Aggressive Consolidation

A bottom-up, metadata-driven approach. Instead of routing through situations or directories, make every practice file self-describing with rich frontmatter, then teach the model to query the frontmatter directly. Combine this with aggressive consolidation: merge the 6 theoretical categories (191 files) into a single `foundations/` directory of ~40-50 files, merge the ~15-20 semantic duplicates in applied categories, and add ~30 coverage-gap files. The result is a smaller, richer corpus (~350 files) where every file carries enough metadata to be found without a router.

- **Architecture shape:** Two-tier directory structure: `foundations/` (~40-50 consolidated theory files) and `applied/` (~250-300 files in ~15 subdirectories, merged from current 19 applied categories). Each file's frontmatter includes: `tags` (3-7 keywords), `applies_when` (1-2 sentence trigger), `priority` (core/situational/niche), `related` (2-3 sibling filenames), `conflicts_with` (principles that create tension). SKILL.md contains no router table and no inlined principles -- just instructions for how to query: "grep frontmatter for tags matching the user's situation, read the top 3-5 by priority."
- **Key seam:** Frontmatter-to-query interface. The model must be able to efficiently scan frontmatter across ~350 files to find relevant practices. This likely requires a compiled manifest (one file listing all frontmatter) that the model reads instead of grepping 350 files individually.
- **Retrieval mechanism:** Bottom-up, metadata-driven. Model reads a compiled `manifest.yaml` or `manifest.md` (~3K tokens: one line per file with tags and applies_when). Model filters by tag/keyword match. Model reads 3-5 matching files. No directory structure needed for routing -- the manifest is the routing layer.
- **Scope of changes:** ~191 theoretical files consolidated into ~45 foundation files. ~15-20 semantic duplicates merged. ~27 exact duplicates removed. ~30-50 new coverage-gap files created. All surviving files (~350) get enriched frontmatter. One new manifest file created. SKILL.md completely rewritten. Total: every file is touched, but the net count drops from 486 to ~350.
- **Failure surface:** The manifest is an even heavier maintenance liability than Option 1's index -- it must include structured data for every file, and drift makes the whole system unreliable. The model's ability to parse a 3K-token manifest and extract the right 3-5 files is unproven at scale. Aggressive consolidation of theoretical categories risks losing nuance (merging 52 Gestalt files into ~10 foundations files requires real editorial judgment). The frontmatter schema must be well-designed upfront -- a bad schema wastes the entire enrichment effort.
- **Prerequisite changes:** Must design the frontmatter schema. Must define the tag vocabulary (a controlled vocabulary, not freeform tags). Must create the consolidation mapping for theoretical categories. Must build the manifest generation script. Must define the new applied-category boundaries (merging from 19 to ~15 subdirectories).
- **Rollback cost:** High. Theoretical category consolidation is destructive (files are rewritten, not just moved). Frontmatter enrichment is additive and harmless to revert. The manifest can be deleted. But restoring the original 191 theoretical files from the consolidated 45 requires git history.
- **Explicit disqualifiers:** Disqualified if 3K tokens for a manifest scan is considered too expensive per invocation (it is higher than the current ~7K only if practice files are also loaded -- but the manifest replaces the router + principles, so net cost depends on how many practice files get loaded). Disqualified if the team cannot invest in building and maintaining the manifest generation pipeline. Disqualified if the model proves unable to reliably extract relevant entries from a dense structured manifest (this needs testing before committing).
